# Phase 2 — 브랜치 생성 + 커밋

## 역할
Phase 1 분석 결과를 바탕으로 브랜치명과 커밋 메시지를 자동 생성하고 즉시 실행한다.

---

## Step 0: Phase 1 Output 검증

Phase 2 시작 전 아래 필드가 모두 있는지 확인한다.

**필수 필드:**
- 프로젝트명
- commit type (feat/fix/refactor/docs/chore 중 하나)
- 변경 파일 목록 (1개 이상)
- 현재 브랜치
- 브랜치 전략 (새 브랜치 / 현재 브랜치 유지)

누락 필드가 있으면 즉시 중단:
```
⚠️ Phase 1 분석 데이터가 불완전합니다.
누락 필드: {누락 필드 목록}

처음부터 다시 실행해주세요.
```

---

## Step 1: 브랜치명 자동 생성

Phase 1에서 "새 브랜치 생성"으로 결정된 경우에만 실행. "현재 브랜치 유지"면 스킵.

| commit type | 브랜치 prefix |
|-------------|-------------|
| feat / refactor / docs / chore | `feature/` |
| fix | `bugfix/` |

- 소문자·하이픈만 사용
- 형식: `{prefix}/{간략한-설명}` (단어 3개 이하)
- 사용자에게 묻지 않고 자동 생성

---

## Step 2: 커밋 메시지 자동 생성

**형식:**
```
{type}: {한국어 설명}

{body}
```

- type은 영어, 설명은 한국어 (클래스명·메서드명·파일명 등 기술 용어 제외)
- body는 모든 타입에서 필수

**Body 포함 항목:**
- 변경의 근본 원인 또는 동기
- 기존 동작 vs 새 동작
- 영향 범위 (파일, 함수, 모듈)

**예시:**
```
fix: 잔고 조회 실패 시 무한 루프 차단

잔고 API 응답 실패 시 예외가 무시되어 동일 요청이 반복됨.
scalper.py의 sync_balance()에서 재시도 횟수 제한 없이
루프가 지속되는 문제.

수정: 최대 3회 재시도 후 포지션 동기화를 중단하고
에러 로그 기록. ScalpingState.desync_count 초기화 추가.
```

## Step 3: 자가 검증 (실행 전 필수)

하나라도 실패하면 교정 후 재확인한다.

**브랜치명 검증:**
- [ ] prefix가 `feature/` 또는 `bugfix/` 중 하나
- [ ] 설명부는 소문자·숫자·하이픈만 사용 (대문자, 공백, 특수문자 없음)

**커밋 메시지 검증:**
- [ ] `{type}: {설명}` 형식 준수
- [ ] type이 `feat/fix/refactor/docs/chore/merge` 중 하나
- [ ] 설명이 한국어 (기술 용어 제외)
- [ ] body가 존재하며 변경 이유와 영향 범위를 실질적으로 설명함

## Step 4: 실행

### 4-0. Worktree 환경 감지

```bash
git rev-parse --git-common-dir
```

- 결과가 `.git`과 다른 경로 → worktree 환경: 브랜치 생성 스킵, Step 4-2로
- 결과가 `.git` → 일반 환경: Step 4-1부터 진행

### 4-1. 브랜치 생성 (일반 환경)

```bash
git checkout -b {브랜치명}
```

⚠️ `git checkout develop`이나 `git pull`을 먼저 실행하지 않는다.
사용자가 develop 브랜치에서 코드를 수정한 상태일 수 있다.
`git checkout -b`는 현재 변경사항을 유지하며 브랜치를 생성하므로 안전하다.

### 4-2. Stage + Commit

```bash
git add {파일들}

git commit -m "$(cat <<'EOF'
{subject line}

{body}
EOF
)"
```

body가 있는 커밋은 항상 HEREDOC을 사용한다. `-m "..."` 직접 전달 시 줄바꿈이 소실된다.

⚠️ 이 단계에서 VERSION 파일을 수정하지 않는다. 버전 업데이트는 Phase 3에서 수행한다.

## 실패 처리

| 명령어 | 최대 재시도 | 재시도 조건 | 재시도 불가 시 |
|--------|-----------|-----------|-------------|
| `git checkout -b` | 1 | 브랜치 이미 존재 → 이름 끝에 `-2` 추가 | 사용자에게 브랜치명 확인 요청 |
| `git add` | 1 | 파일 없음 → `git status` 재확인 후 목록 갱신 | 누락 파일 사용자에게 보고 |
| `git commit` (nothing to commit) | 0 | 재시도 없음 — Phase 1 분석 오류 | 워크플로우 종료, Phase 1 재실행 안내 |
| `git commit` (pre-commit hook 실패) | 1 | hook 출력 확인 후 수정, 재시도 | hook 오류 사용자에게 보고 |
